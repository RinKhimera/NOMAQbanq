### ⚠️ Gate & environnement (leçons de F1 — CRITIQUE)

- **Ne lance JAMAIS `bun run check`** ni `prettier --check .` / `prettier --write .`. Le repo a un **CRLF working-tree pré-existant** sur ~200 fichiers (l'index est en LF, la CI Linux passe) → `prettier --check` échoue localement et n'est PAS un signal de correction. **Gate réel = `bun run type-check` + `bun run lint`** (les deux à 0). Lance `bunx prettier --write` **uniquement sur les fichiers que tu modifies**.
- **`bun run test`** = tests frontend (happy-dom). **`bun run test:integration`** = crée une branche Neon éphémère (via `.env.local`), migre, lance TOUT le projet vitest `integration`, détruit la branche. Le filtre `-- <fichier>` **ne marche pas** (les args n'atteignent pas vitest) → ça lance toute la suite ; **lance-le avec parcimonie** (chaque run = une branche Neon). 
- **Échec d'intégration connu, pré-existant et SANS lien : `tests/integration/uploads-actions.test.ts`** (CDN `cloudfront.net` vs `cdn.nomaqbanq.ca`). Baseline = **157/158**. Ne le compte pas comme une régression.
- **Migrations Drizzle :** `bun run db:generate` → inspecte le SQL généré → `bun run db:migrate` (cible la base Neon de dev via `DATABASE_URL_UNPOOLED` ; l'utilisateur a autorisé les migrations dev). Vérifie qu'aucun `DROP` involontaire n'apparaît avant de migrer.
- **Commits :** pour un cluster couplé, tsc peut être rouge en cours — ne committe pas un état où `tests/integration` ne peut même pas s'importer ; l'**état final du cluster doit être vert** (type-check + lint + test + test:integration). Sinon, committe chaque tâche seulement si `bun run type-check` passe.
- **Worktrees d'agents :** si `.claude/worktrees/**` apparaît, c'est un résidu — `git worktree remove --force` + `git worktree prune` (eslint les ignore déjà via `eslint.config.mjs`).
- **Rapports de revue** (`docs/superpowers/reviews/`) = **jetables** : supprime-les après triage, ne les committe jamais.

## C4 — Fiabilité paiements & accès (#111)

### Constats (confirmés dans le code)

1. **`refunded → completed` ne re-crédite pas** : `updateManualTransaction`
   (`features/payments/actions.ts:201-206`) ne déclenche `revokeAccessIfLast` que sur
   `completed → refunded` ; la transition inverse ne restaure jamais l'accès.
2. **`revokeAccessIfLast` efface le cumul** : `features/payments/lib.ts:156-164` fait un
   `DELETE` de la ligne `userAccess` alors que des transactions antérieures avaient
   étendu la fenêtre. Chaque transaction porte son `accessExpiresAt` (snapshot du cumul
   à l'octroi) : l'information pour restaurer existe.
3. **Parseur de montants dupliqué** : `edit-transaction-modal.tsx:84-85` réimplémente
   `parseAmountToCents` (`lib/currency.ts:14`).

### Design : `recomputeAccess` (approche retenue — recompute dérivé des transactions)

Fonction unique `recomputeAccess(tx, userId, { excludeTransactionId? })` dans
`features/payments/lib.ts`, remplaçant `revokeAccessIfLast` :

- **Verrou** : même `SELECT user … FOR UPDATE` que `grantManualAccess` (sérialise tous
  les octrois/révocations d'un même utilisateur — convention data-layer).
- **Calcul par type** (`exam`, `training`) : `expiresAt = max(accessExpiresAt)` des
  transactions `completed` de l'utilisateur couvrant ce type — `accessType` égal **ou**
  produit combo (join `products.isCombo`) ; `excludeTransactionId` ignoré du calcul
  (cas suppression). `lastTransactionId` = la transaction porteuse du max.
- **Aucune transaction ne couvre le type** → suppression de la ligne `userAccess`.
- **`expiryReminderSentAt`** : remis à `null` uniquement si l'expiration avance
  (sémantique identique à `grantManualAccess`).
- **Appelants** :
  - `updateManualTransaction` : recompute dès que le statut change (couvre les deux
    sens `completed ↔ refunded` et tout futur statut).
  - `deleteManualTransaction` : recompute **avant** le `DELETE` de la transaction, avec
    `excludeTransactionId` — la FK `userAccess.lastTransactionId` est `NOT NULL` +
    `onDelete: restrict`, la ligne d'accès doit être re-pointée/supprimée d'abord.
- **Hors périmètre** : `grantManualAccess` et le webhook Stripe (chemins d'octroi) ne
  changent pas.

**Sémantique assumée** : restaurer `max(accessExpiresAt)` des transactions restantes
redonne l'expiration telle qu'elle était avant la transaction retirée. Un recalcul
« comme si la transaction n'avait jamais existé » (re-simulation du cumul) donnerait un
résultat différent quand une transaction intermédiaire est retirée — hors scope,
documenté dans le JSDoc de la fonction.

### Item 3 — modal

`edit-transaction-modal.tsx` importe `parseAmountToCents` / la validation de
`lib/currency` et supprime son parseur local.

### Tests d'intégration (`tests/integration/`, `bun run test:integration`)

- Cumul non-combo : 2 transactions, revoke de la dernière → restauration de
  l'expiration de la première.
- Transaction unique revoked → ligne `userAccess` supprimée.
- `refunded → completed` → re-crédit de l'accès.
- Combo : recompute couvre les deux types.
- `deleteManualTransaction` : avec accès pointé (re-pointage/suppression) et sans.
- Idempotence : deux recomputes successifs = même état.
- Cleanup `afterAll` dans l'ordre des FK (enfants avant parents).
